COMPLETE PROMPT
============================================================

Create a **single-file, sole client-side JavaScript application** for depicting, editing, and animating an event-based message flow between components of a software system.

The application must run entirely in the browser using HTML, CSS, and JavaScript. It must not require a backend, server-side storage, build step, npm installation, or external paid services. Prefer plain JavaScript and browser-native APIs. External open-source libraries may be used only when loaded via CDN and clearly justified.

## 1. Application Goal

The application shall allow users to visually model a software system as components connected by event/message flows.

Users shall be able to:

- Draw and edit components.
- Connect components with directed message flows.
- Define the execution sequence of the message flows.
- Define message names.
- Define processing actions performed by components.
- Upload processing images for individual flow steps.
- Edit the flow through both the drawing canvas and a structured flow panel.
- Animate the defined flow in presentation mode.

The application should feel similar to a lightweight draw.io-style editor focused specifically on event-based component flows.

## 2. Core Concepts and Naming

Use the following consistent domain model:

- **Component**: A visual node representing a software system component, service, module, application, database, external system, or actor.
- **Message Flow**: A directed connection between two components.
- **Flow Step**: One execution step in the sequence.
- **Message**: The event or information sent from one component to another.
- **Processing Action**: The action performed by the target component after receiving a message.
- **Processing Image**: An optional image associated with a flow step and shown during animation.
- **Flow Panel**: A compact, list-based representation of the ordered flow steps.
- **Canvas**: The visual drawing area.
- **Presentation Mode**: A focused mode for running the animation with maximum visibility.

Each **Flow Step** shall contain:

- Sequence number.
- Source component.
- Target component.
- Message name or message text.
- Processing action text.
- Optional processing image.
- Optional notes.

## 3. General Application Requirements

The application shall provide:

- A full-page browser-based user interface.
- A central canvas for drawing and editing components and message flows.
- A compact top toolbar with icons and grouped controls.
- A flow panel showing the ordered sequence of flow steps.
- A properties panel or inline editing mechanism for selected elements.
- A presentation mode for animation.
- A clean, modern, readable interface.
- Mouse and keyboard interaction for efficient editing.
- Clear visual feedback for selection, hover, drag, resize, connection creation, editing, and animation state.

## 3.1 Canvas Navigation and Interaction Modes

The canvas shall support easy navigation for larger diagrams.

### Zooming

Requirements:

- Zooming must be easy, discoverable, and usable during normal editing.
- Provide visible zoom controls in the toolbar, including zoom in, zoom out, reset to 100%, and fit to screen.
- Provide a zoom percentage display or selector.
- Support mouse-wheel or trackpad zooming using a common modifier key such as `Ctrl/Cmd + wheel`.
- Support trackpad pinch zoom where the browser and device expose it reliably.
- Support keyboard shortcuts for zooming where feasible, for example `Ctrl/Cmd + +`, `Ctrl/Cmd + -`, and `Ctrl/Cmd + 0`.
- Zooming should be centered around the mouse pointer when triggered from the canvas, so users can zoom into the area they are looking at.
- Component labels, message labels, resize handles, and animation symbols must remain usable and readable at reasonable zoom levels.

### Panning

Requirements:

- Panning must be possible in a draw.io-like way.
- The application shall explicitly distinguish between **selection mode** and **pan / hand mode**.
- In selection mode, left mouse drag on an empty canvas shall create a selection rectangle, and left mouse drag on a selected element shall move it.
- In pan / hand mode, left mouse button + mouse drag shall pan the canvas viewport instead of selecting or moving elements.
- The active mode shall be clearly visible in the toolbar and through the cursor style.
- Users shall be able to switch between selection mode and pan / hand mode from the toolbar.
- Optional convenience shortcuts may be provided, such as holding `Space` to temporarily pan in editing mode, using the middle mouse button to pan, or using trackpad gestures to pan.
- Panning shall not accidentally move components or start connections while pan / hand mode is active.
- Pan and zoom state should be persisted where useful and restored on reload.

## 7. Message Flow Editing Requirements

The application shall allow users to define directed message flows between components.

### 7.1 Message Flow Creation

Requirements:

- Users shall be able to connect a source component to a target component with an arrow.
- The arrow shall represent a message flow.
- The message flow shall have a message name or message text.
- The message flow shall have a sequence number.
- Sequence numbers shall define the execution order of the animation.
- The sequence number shall be displayed near the arrow in editing mode.
- The application shall support multiple message flows between the same components.
- The application shall support self-referencing flows where source and target are the same component, if technically feasible.

### 7.2 Message Flow Editing

Users shall be able to:

- Rename or edit the message directly in the drawing area by double-clicking the message label.
- Rename or edit the message from the flow panel.
- Edit the sequence number.
- Drag connection labels, where feasible.
- Delete message flows.
- Reconnect a message flow to another source or target component.
- Change arrow color.
- Change arrow thickness.
- Configure the connection style.
- Automatically update arrow positions when connected components are moved or resized.

### 7.3 Configurable Connection Style

The application shall allow users to configure the visual style of message flow connections.

Supported connection styles:

- **Straight**: a direct straight line between source and target component.
- **Arc / Curved**: a smooth curved connection.
- **Angular / Elbow**: an orthogonal connection with right-angled segments.

Requirements:

- Each message flow shall store its own connection style.
- Users shall be able to change the connection style for a selected message flow.
- Users shall be able to set a default connection style for newly created message flows.
- The selected connection style shall be reflected immediately on the canvas.
- Message animation shall follow the actual rendered path of the selected connection style.
- Arrowheads must remain correctly oriented for all connection styles.
- Moving or resizing connected components shall automatically update the connection path.
- Multiple connections between the same components should remain visually distinguishable, especially when using curved or arc styles.

Optionally support:

- Bend points.
- Curvature.
- Label position.

### 7.4 Message Label Display

The application shall distinguish between editing mode and animation mode.

In editing mode:

- Message names shall be visible on the canvas by default.
- Sequence numbers shall be visible on the canvas by default.

In animation mode:

- Do not display all message names permanently on the arrows.
- Only the currently animated message shall be displayed.
- The currently animated message name shall be shown near or underneath the moving message symbol.
- Non-active messages shall not clutter the animation view.
- Sequence numbers may be hidden or minimized during presentation mode unless needed for orientation.

## 8. Processing Action Requirements

- Processing action text shall support multiline text.

## 9. Processing Image Requirements

Requirements:

- Supported image formats should include PNG, JPG, JPEG, GIF, SVG, and WebP where browser-supported.
- The uploaded image shall be stored client-side as part of the application state, preferably as a data URL.
- Users shall be able to preview the uploaded processing image.
- Users shall be able to replace or remove the processing image.
- During animation, the processing image of the currently active flow step shall be displayed in a dedicated image display area on the right side of the screen.
- The processing image shall remain visible until the next animation step starts.
- The image display area shall scale the image proportionally without distortion.
- If no processing image exists for the current step, the image area shall show an empty state or remain hidden.

## 10. Flow Panel Requirements

The application shall provide a compact list-based flow panel showing the ordered message flow sequence.

### 10.1 Default Compact View

The flow panel shall display flow steps compactly by default.

All non-selected flow steps shall remain collapsed and compact. The collapsed representation shall use minimal vertical space so that many ordered steps remain visible at once.

Each collapsed flow step item shall show only the most important information:

- Sequence number.
- Source component name.
- Target component name.
- Message name or message text.
- Indicator if a processing action exists.
- Indicator if a processing image exists.

The compact view should make it possible to see many flow steps at once without excessive scrolling.

### 10.2 Expandable Details

Each flow step item shall be expandable.

When expanded, the flow step shall show and allow editing of:

- Sequence number.
- Source component.
- Target component.
- Message name or message text.
- Processing action text.
- Processing image preview.
- Processing image upload, replace, and remove controls.
- Optional notes.

The flow panel shall behave as an accordion. Exactly one flow step may be expanded at a time. Selecting a flow step shall expand that selected step. All other flow steps shall automatically collapse. Clicking or selecting another step shall move the expansion to the newly selected step. Collapsed items shall remain compact and readable. The currently selected or animated flow step may be expanded automatically when useful, but this must not interrupt active text editing.

### 10.3 Stable Inline Editing

The flow panel must support reliable inline editing.

Requirements:

- Text fields in the flow panel must remain editable without losing focus after each keystroke.
- Editing a field must not trigger a full re-render of the flow panel that replaces the active input element.
- While the user is typing, the cursor position, focus, and partially entered text must be preserved.
- Updates to the application state should happen in a way that does not interrupt typing, for example:
  - update state on `input` without replacing the focused DOM node,
  - commit changes on `blur` or `Enter`,
  - or use a local edit buffer while the field is focused.
- The following fields must be editable in the flow panel:
  - message text
  - processing action text
  - notes
  - sequence number
  - component references, where supported
- Editing in the flow panel must synchronize with the canvas immediately or after the edit is committed.
- The accordion behavior must not collapse or re-create the currently edited item while the user is typing.
- The implementation must explicitly avoid the bug where text fields lose focus after every keystroke.

### 10.4 Flow Panel Editing

The flow panel shall allow users to:

- Edit message text directly in the list.
- Edit processing action text directly in expanded details.
- Upload, replace, preview, or remove the processing image.
- Reorder flow steps by drag and drop.
- Automatically update sequence numbers after reordering.
- Select a flow step and highlight the corresponding message flow on the canvas.
- Select a message flow on the canvas and highlight the corresponding flow step in the panel.
- Delete a flow step.
- Add a new flow step, where useful.

### 10.5 Synchronization

The canvas and flow panel shall always stay synchronized.

Requirements:

- Renaming a component on the canvas shall update the flow panel.
- Editing a message in the flow panel shall update the arrow label on the canvas.
- Editing a message by double-clicking it on the canvas shall update the flow panel.
- Reordering flow steps in the flow panel shall update sequence numbers on the canvas.
- Deleting a message flow on the canvas shall remove the corresponding flow step from the flow panel.
- Deleting a component shall also delete or clearly handle affected message flows.

## 11. Animation Semantics

The application shall animate the event-based message flow according to the defined sequence of flow steps.

### 11.1 Animation Behavior

For each flow step:

1. The source component shall be visually highlighted.
2. A message symbol shall move from the source component to the target component along the selected connection path.
3. Only the currently animated message text shall be displayed near or underneath the moving message symbol.
4. When the message reaches the target component, the message symbol shall remain visible at the target component.
5. The application shall wait for user confirmation before hiding the arrived message and showing the processing action.
6. After the user clicks or presses the next-step control, the message symbol shall disappear.
7. The target component shall be highlighted.
8. The processing action text shall be displayed on top of or above the target component.
9. The processing image associated with the current flow step shall be displayed in the image display area.
10. The application shall then continue to the next flow step or wait for user interaction, depending on the selected animation mode.

### 11.2 Message Arrival Interaction

The animation shall explicitly distinguish between two phases of a flow step.

#### Phase 1: Message Transfer

- The message moves from the source component to the target component.
- The message text is visible during movement.
- When the message arrives, it remains visible at the target component.
- The processing action is not shown yet.

#### Phase 2: Processing Action

- Triggered by user click, next-step button, or configured auto-continue behavior.
- The arrived message disappears.
- The processing action text appears on or above the target component.
- The processing image appears in the right-side image area.
- The target component is highlighted as actively processing.

In step-by-step mode, the transition from message arrival to processing action shall always wait for user input.

In auto play mode, the application may either:

- wait for user input at message arrival, or
- provide a configurable auto-continue delay.

The default behavior shall be to wait for user input at message arrival, so the presenter can explain the message before the processing action appears.

### 11.3 Visual Animation Requirements

The animation shall provide:

- A clearly visible moving message symbol.
- A readable label only for the currently animated message.
- Highlighting of the active source component.
- Highlighting of the active target component.
- Highlighting of the active arrow/message flow.
- A visible processing state when the target component performs its action.
- Smooth animation transitions.
- Configurable animation speed.

### 11.4 Animation State

The application shall track animation state, including:

- Current flow step.
- Current phase of the step:
  - message transfer
  - message arrived
  - processing action
- Current animation mode.
- Playing, paused, stopped, or completed state.
- Previously completed steps.
- Active message flow.
- Active source component.
- Active target component.
- Active processing action.

The application shall allow users to configure the speed of the message flow animation.

Requirements:

- Users shall be able to choose the message movement speed.
- Provide simple predefined speed options:
  - Slow
  - Normal
  - Fast
- Optionally provide a slider for fine-grained speed control.
- The selected speed shall affect the duration of the moving message animation.
- The selected speed shall be used consistently in both normal animation mode and presentation mode.
- The selected speed should be persisted in local storage.

### 12.3 Animation Modes

The application shall support the following animation modes.

#### Auto Play Mode

- The application automatically executes the flow steps in sequence.
- The user can pause, resume, stop, or restart the animation.
- Message movement speed shall follow the selected speed configuration.
- At message arrival, the default behavior shall be to pause and wait for user confirmation before showing the processing action.
- Optionally, the user may enable auto-continue after message arrival with a configurable delay.

#### Step-by-Step Mode

- The application executes one phase at a time.
- The first click starts the message transfer.
- The next click after arrival hides the message and shows the processing action.
- The next click proceeds to the next flow step.
- The user can move backward to the previous phase or previous step.
- The current step and phase shall remain visible until the user advances.

## 13. Presentation Mode Requirements

The application shall provide a presentation mode optimized for visibility.

Requirements:

- Presentation mode shall maximize the canvas area.
- Non-essential editing controls shall be hidden or minimized.
- The background shall be lean and clean.
- The grid shall be hidden in presentation mode, even if it is enabled in editing mode.
- No grid lines, editing guides, snap indicators, resize handles, selection boxes, or other editing artifacts shall be visible in presentation mode unless actively needed for animation.
- Animation controls shall remain accessible.
- The top toolbar shall be simplified in presentation mode.
- Component names and currently active message text shall remain readable.

- Only the currently animated message shall be displayed.
- The active processing image shall be displayed prominently in a fixed presentation image panel on the right side.
- The presentation image panel shall be closable or collapsible by the presenter.
- When the presentation image panel is open, the canvas layout shall reserve space for it where feasible, so the panel does not cover or hide components.
- When the presentation image panel is closed, the canvas shall use the freed space for the diagram.
- The panel shall provide a clear reopen control when closed.
- If no processing image exists for the current step, the panel may show an empty state or be collapsed, depending on the selected behavior.
- The layout shall be suitable for screen sharing or presenting in meetings.
- Users shall be able to exit presentation mode easily.

### 13.1 Presentation Mode Connection Visibility

Presentation mode shall reduce visual clutter while keeping the overall flow structure visible.

Requirements:

- In presentation mode, the canvas background shall remain plain and uncluttered, with no grid visible.
- In presentation mode, all non-active connections shall remain visible but be shown with high transparency.
- Non-active connections shall be barely visible, for example using low opacity.
- Only the currently animated message flow shall be highlighted prominently.
- The active message flow shall use a strong visual style, for example:
  - higher opacity
  - thicker line
  - highlighted color
  - visible arrowhead
- Non-active message names shall not be shown during animation.
- Only the currently animated message name shall be displayed near the moving message symbol.
- Components may remain visible normally, but only the active source and target components shall be highlighted.
- Completed steps may optionally receive a subtle completed-state indicator, but this must not visually compete with the active step.

### 13.2 Presentation Mode Processing Image Panel

Presentation mode shall include a dedicated processing image panel.

Requirements:

- The processing image panel shall be a fixed right-side section of the presentation layout.
- The panel shall be visually separated from the canvas and shall not behave like a floating overlay that covers diagram elements.
- The panel shall scale images proportionally without distortion.
- The panel shall support PNG, JPG, JPEG, GIF, SVG, and WebP images where browser-supported.
- The presenter shall be able to close or collapse the panel at any time.
- Closing the panel shall immediately free the horizontal space for the canvas, or at minimum ensure that no component is hidden behind the panel.
- The presenter shall be able to reopen the panel through a clearly visible compact control.
- The panel state may persist during the current presentation session.
- If a flow step has no processing image, the panel shall show a subtle empty state or remain collapsed without disrupting the animation.
